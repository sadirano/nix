# v0.10.0 release checklist

Manual verification for promoting `v0.10.0-pre` to the official `v0.10.0`.
Everything here is Windows-only (the supported platform).

**Note:** `v0.10.0-pre2` was 12 commits stale (missing `[notify]`, `[bin]`
exports, multi-name `[shortcuts]`, and the wrapper rename-aside fix) —
`v0.10.0-pre3` was cut from HEAD (`97cc276`) to test against instead.
Treat `-pre3` as the effective candidate wherever this file says `-pre`.

**Sandboxing:** steps marked 🧪 must run against a scratch store, never the real
one. In a fresh PowerShell:

```powershell
$env:NIX_HOME = "$env:TEMP\nix-rc"        # scratch store for this shell only
```

Steps touching the **real** `~/.nix` are marked ⚠️ — snapshot first:

```powershell
Copy-Item ~/.nix ~/.nix-pre-0.10-backup -Recurse
```

## 2. `--init` / PATH persistence (breaking: no pwsh snippet, no `--skip-profile`)

- [x] 🧪 In a scratch shell: `nix --init` completes; **verify `$PROFILE` is
      untouched** (compare timestamps/content before and after).
      (SHA256 + LastWriteTime identical before/after.)
- [x] 🧪 `~/.nix/bin` (scratch) is on the *user* PATH in the registry
      (`Get-ItemProperty HKCU:\Environment -Name Path`).
      **Cleanup when done testing** — the scratch entry persists in the real
      registry; remove exactly it (type-preserving, no `setx`):

      ```powershell
      $bin = "$env:TEMP\nix-rc\bin"
      $k = Get-Item HKCU:\Environment
      $raw = $k.GetValue('Path','','DoNotExpandEnvironmentNames')
      $new = ($raw -split ';' | Where-Object { $_ -ne $bin }) -join ';'
      Set-ItemProperty HKCU:\Environment -Name Path -Value $new -Type $k.GetValueKind('Path')
      ```
      (done — cleaned up.)
- [x] 🧪 All wrapper exes exist after init: `o e s y p r sg ff nix`.
- [x] 🧪 `~/.nix/AGENTS.md` is (re)generated by `--init`/`--sync`.
- [x] 🧪 No `~/.nix/shell/` dir is created on Windows; with a pre-existing
      `shell/nix.ps1`, `--sync` **deletes it** and prints the removal note
      (the `$PROFILE` dot-source warning + the `function q { exit }` tip).
- [x] 🧪 `nix --init --skip-profile` now errors as an unknown flag.
- [x] 🧪 `nix --doctor` shows no `shell` row on Windows; `wrappers`/`PATH`
      rows unchanged.
- [ ] A brand-new terminal (not a refreshed one) picks up the PATH and `o`/`r` work.
      **Needs you** — open a fresh terminal window (not this session) and
      confirm `o`/`r` resolve. Everything else in this section is verified.

## 3. `--resolve` is read-only (breaking: no directory creation)

- [x] 🧪 Register an alias, `rmdir` its directory, then `nix <alias>`:
      resolves (exit 0, prints the path) but **does not re-create the
      directory**. (Wording fix: the actual/tested contract is "never
      re-creates," not "errors" — matches e2e.zig:479, checked exactly.)
- [x] 🧪 Same for a segmented form (`nix docs@<alias>`): no directory created.
- [x] 🧪 Registering a *new* alias (`nix name path`) still auto-creates the dir
      (creation moved to registration, not resolution).

## 4. `--export` / `--import` portable backup

- [x] 🧪 `nix --export backup.toml` writes a file containing aliases + groups.
- [x] 🧪 Delete one alias, `nix --import backup.toml`: **merge** restores the
      missing alias without overwriting ones that changed since the export.
- [x] 🧪 `nix --import --replace backup.toml`: store matches the export exactly.
- [x] ⚠️ Export the real store once and stash the file with the `~/.nix` snapshot
      (doubles as the rollback artifact for this release).

## 5. `--which` reverse lookup

- [x] `nix --which <alias-dir>` prints the alias.
- [x] `nix --which <alias-dir>\deep\nested\file` resolves to the same alias.
- [x] With nested roots (one alias inside another), the **deepest** alias wins.
- [x] `nix --which C:\Windows` (outside every alias) errors, empty stdout.
- [x] Bare `nix --which` uses the cwd.

## 6. Child-process environment

- [x] `r <alias> pwsh -c 'echo $env:NIX_ALIAS $env:NIX_ALIAS_PATH'` shows the
      alias name and absolute path.
- [x] Same variables visible inside a saved action (`r <alias> :<name>`) and a
      `.nix/scripts/` script.
      **Bug found + fixed:** a bare-name direct run of a `.nix/scripts/*.ps1`
      (`r <alias> foo` resolving `foo.ps1`) failed with `InvalidExe` —
      CreateProcess can't launch `.ps1` without an interpreter, unlike the
      `.cmd`/`.bat`/`.exe` candidates the same probe also produces. Fixed by
      routing a resolved `.ps1` through `pwsh`/`powershell -File` (`run.zig`'s
      new `wrapPs1`, sharing `bin_exports.zig`'s `.ps1` trampoline logic via a
      new `proc.psShell`). Rebuilt, re-synced, full test+e2e (73 checks) green,
      re-verified against the daily install.
- [x] `r <alias> pwsh -c 'echo $env:FZF_DEFAULT_OPTS'` is **empty** — the
      picker options must not leak into user subshells. (Verified with the
      ambient var cleared first — it was already set in this shell's own
      profile, unrelated to nix.)
