# v0.10.0 release checklist

Manual verification for promoting `v0.10.0-pre` to the official `v0.10.0`.
Everything here is Windows-only (the supported platform).

**Note:** `v0.10.0-pre2` was 12 commits stale (missing `[notify]`, `[bin]`
exports, multi-name `[shortcuts]`, and the wrapper rename-aside fix) —
`v0.10.0-pre3` was cut from HEAD (`97cc276`) to test against instead.
Treat `-pre3` as the effective candidate wherever this file says `-pre`.

**Sandboxing:** steps marked 🧪 must run against a scratch store, never the real
one. In a fresh PowerShell:

```powershell
$env:NIX_HOME = "$env:TEMP\nix-rc"        # scratch store for this shell only
```

Steps touching the **real** `~/.nix` are marked ⚠️ — snapshot first:

```powershell
Copy-Item ~/.nix ~/.nix-pre-0.10-backup -Recurse
```

      **Cleanup when done testing** — the scratch entry persists in the real

      ```powershell
      $bin = "$env:TEMP\nix-rc\bin"
      $k = Get-Item HKCU:\Environment
      $raw = $k.GetValue('Path','','DoNotExpandEnvironmentNames')
      $new = ($raw -split ';' | Where-Object { $_ -ne $bin }) -join ';'
      Set-ItemProperty HKCU:\Environment -Name Path -Value $new -Type $k.GetValueKind('Path')
      ```
      (done — cleaned up.)

## 6. Child-process environment

- [x] `r <alias> pwsh -c 'echo $env:NIX_ALIAS $env:NIX_ALIAS_PATH'` shows the

- [x] Same variables visible inside a saved action (`r <alias> :<name>`) and a
      `.nix/scripts/` script.
      **Bug found + fixed:** a bare-name direct run of a `.nix/scripts/*.ps1`
      (`r <alias> foo` resolving `foo.ps1`) failed with `InvalidExe` —
      CreateProcess can't launch `.ps1` without an interpreter, unlike the
      `.cmd`/`.bat`/`.exe` candidates the same probe also produces. Fixed by
      routing a resolved `.ps1` through `pwsh`/`powershell -File` (`run.zig`'s
      new `wrapPs1`, sharing `bin_exports.zig`'s `.ps1` trampoline logic via a
      new `proc.psShell`). Rebuilt, re-synced, full test+e2e (73 checks) green,
      re-verified against the daily install.
- [x] `r <alias> pwsh -c 'echo $env:FZF_DEFAULT_OPTS'` is **empty** — the
      picker options must not leak into user subshells. (Verified with the

      profile, unrelated to nix.)

## 10. `[shortcuts]` wrappers

- [x] 🧪 Add a `[shortcuts]` rename (e.g. `g = "sg"`), run `nix --sync`:
      new wrapper exe appears and dispatches correctly via argv0. (Confirmed

      running `sg`/`g` for real opens fzf.)
- [x] 🧪 Rename it again + `--sync`: the **old** wrapper exe is deleted.
      **Bug found + fixed:** a custom name renamed to ANOTHER custom name
      (`sg="g"` then `sg="gg"`) never deleted the old wrapper — only a
      builtin's own name being renamed away was ever cleaned up. Fixed with
      a `~/.nix/wrappers.toml` manifest (see the `fix:` commit). Re-verified
      the exact repro after rebuilding: `gg.exe` now gets deleted on the next
      rename.
- [x] 🧪 An invalid shortcut name is rejected at sync time. (Not a loud
      rejection — `loadConfig` silently drops an unusable custom name at
      parse time via `store.validateAliasName`, so the slot gracefully keeps
      its builtin default (`sg.exe` still works) rather than installing a bad
      wrapper or erroring. Confirmed with `sg = "bad name!"`: sync exits 0,
      no `bad name!.exe` created, `sg.exe` still dispatches correctly.)
- [x] `--sync` while a shell window is open (wrapper exes potentially in use)
      still completes — atomic replace, no half-written exes.
- [x] ⚠️ `--sync` while an `r <alias> <long-cmd>` session is live: ran a
      20s `.cmd` script via `r alpha longrun` in the background, then
      `--sync` mid-flight. Completed in 0.03s with no error; the live session
      ran to completion (`finished` marker file written) untouched; `r.exe`
      was intact and functional afterward. No `.stale` file appeared — on
      this system the plain rename succeeded transparently even against the
      running image (matches the code comment: Windows denies *overwriting*

      no half-written exe — is confirmed either way.
